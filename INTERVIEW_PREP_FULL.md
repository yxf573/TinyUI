# TinyElement 全量面试题库

## 1. 使用方式

这份文档是上一版项目总述的扩展版，重点解决两类问题：

- 面试官按组件逐个追问时，怎么结合真实代码回答。
- 面试官从技术栈角度深挖 Vue、Vite、TypeScript、Pinia、测试、工程化时，怎么系统回答。

建议实际使用时这样分层：

- 先用 [INTERVIEW_PREP.md](./INTERVIEW_PREP.md) 回答项目总述。
- 被追问到具体组件时，翻本文件的“逐组件问答”。
- 被追问到框架原理、构建、状态管理、测试和工程化时，翻本文件的“技术深挖问答”。

## 2. 开场先说清楚的事实

- 这个项目真实使用了 Vue 3、TypeScript、Vite、VitePress、Vue Router、Vitest、async-validator、Popper、Node.js、Qdrant。
- 这个项目没有接入 Pinia。
- 如果面试官问到 Pinia，不能假装“项目里用了”，应该先说明“当前项目主要是组件库，状态大多是组件局部状态和父子通信，所以没有引入全局状态管理”，然后再继续回答“如果后续需要接入，我会怎么设计”。

## 4. 逐组件问答

## 4.1 Button

### 组件定位

Button 是最基础的交互组件，但在组件库里它不是“只有一个按钮”，而是一个要兼顾类型、尺寸、禁用、加载态、图标和原生按钮语义的通用组件。

### 高频问题 1：你怎么设计 Button 的通用能力？

**S**

组件库里的 Button 会被高频复用，如果 props 设计不统一，后续所有页面都会出现使用不一致的问题。

**T**

我要让 Button 既支持常见视觉风格，也保留原生按钮能力，比如 `type`、`size`、`nativeType`、`disabled`、`loading`、`icon`。

**A**

我把这些能力统一定义在 `src/components/Button/types.ts` 里，通过 `buttonProps` 约束按钮类型和尺寸取值；在组件里再通过 `computedClass` 动态生成样式类名，并且让 `loading` 和 `disabled` 一起控制按钮不可点击。为了兼容表单场景，我还保留了原生 `button / submit / reset` 的 `nativeType`。

**R**

这样 Button 不只是样式按钮，而是具备了“可复用 + 可约束 + 可语义化”的基础能力，后续其他组件和 demo 里都可以稳定复用。

### 高频问题 2：Button 为什么要 `defineExpose`？

**S**

在业务里有些场景需要父组件直接拿到按钮实例，比如聚焦、测量尺寸或者做命令式控制。

**T**

我要在不破坏封装的前提下，把真正需要暴露的能力开放给外部。

**A**

我在 Button 里用 `ref<HTMLButtonElement>()` 拿到原生按钮节点，再通过 `defineExpose({ ref: _ref })` 只暴露这个引用，而不是把整个内部状态都暴露出去。

**R**

这样做兼顾了封装性和可用性，体现的是“组件对外暴露最小必要能力”的设计思路。

### 面试加分补充

- Button 的 `loading` 态本质上是行为约束，不只是图标切换。
- `disabled || loading` 一起控制不可点击，避免重复提交。

## 4.2 ButtonGroup

### 组件定位

ButtonGroup 是一个典型的组合型样式组件，它本身逻辑很轻，但对组件库的一致性很重要。

### 高频问题 1：ButtonGroup 这种简单组件为什么还值得保留？

**S**

如果只靠业务页面自己写相邻按钮样式，样式规则容易散落在页面里，复用性很差。

**T**

我要把“相邻按钮分组展示”抽象成一个独立组件，统一控制布局和视觉效果。

**A**

我把 ButtonGroup 设计成一个纯容器组件，主要通过 slot 承载多个按钮，再由对应样式文件统一处理边框衔接、圆角和间距。

**R**

这样业务层只需要关心放哪些按钮，不需要重复处理组合样式，组件库的一致性也更好。

### 高频问题 2：简单组件怎么体现你的抽象能力？

**S**

面试里简单组件最容易被忽略，但它其实能体现你会不会做“职责最小化”。

**T**

我要让 ButtonGroup 只管组合，不去侵入 Button 的内部行为。

**A**

我让它只渲染 slot 内容，不接管子按钮的点击、禁用、加载等状态，把行为责任全部留给 Button 本身。

**R**

这说明我在组件设计上会尽量让每个组件只负责一件事，而不是把样式、行为和状态都耦合到一起。

## 4.3 Alert

### 组件定位

Alert 用于页面内提示，强调持续可见、语义明确和关闭能力。

### 高频问题 1：Alert 和 Message 的区别是什么？

**S**

很多面试官会用 Alert 和 Message 来判断你对反馈类组件的场景区分是否清楚。

**T**

我要明确区分“页面内常驻提示”和“全局临时浮层提示”。

**A**

Alert 是声明式组件，通常直接写在页面结构里，适合持续展示的提示信息；Message 是函数式组件，适合顶部浮层通知、几秒后自动消失。代码上 Alert 主要是 `visible` 控制和 `close` 事件，Message 则是动态创建、实例管理和自动销毁。

**R**

这样我在回答时就不会把反馈类组件混在一起，而是能说明不同交互模型背后的实现差异。

### 高频问题 2：Alert 的关闭设计你是怎么做的？

**S**

Alert 的关闭看似简单，但它既涉及 UI 状态，也涉及对外通知。

**T**

我要让组件内部正确隐藏，同时把关闭动作通知外部。

**A**

组件内部通过 `visible` 管理显示状态，点击关闭按钮后先把 `visible` 设为 `false`，再通过 `emit('close', evt)` 抛出关闭事件。为了支持不同场景，我还补了 `closable`、`closeText`、`showIcon`、`effect`、`center` 等可配置项。

**R**

最终 Alert 既能满足简单提示，也能支持图标、主题和自定义关闭入口，扩展性更好。

## 4.4 Container / Header / Footer / Main / Aside

### 组件定位

Container 负责布局容器抽象，重点不是业务逻辑，而是如何让页面布局更声明式、更稳定。

### 高频问题 1：Container 的方向为什么要支持自动推断？

**S**

如果每次使用都强制传 `direction`，会增加模板冗余，而且用户很容易漏传。

**T**

我要让 Container 在常见布局场景下具备“合理默认行为”。

**A**

我在 `Container.vue` 里结合 `useSlots()` 读取默认插槽中的子节点，如果发现子组件名里有 `ElHeader` 或 `ElFooter`，就自动判定为纵向布局；如果显式传了 `direction`，再以 props 为准。

**R**

这样组件兼顾了灵活性和易用性，既能手动指定，也能在典型布局下自动推断，使用体验更像成熟组件库。

### 高频问题 2：为什么把 Header/Footer/Main/Aside 做成独立组件？

**S**

如果只是写一个大容器，布局结构不够语义化，也不利于统一样式。

**T**

我要把布局拆成可组合的语义块。

**A**

我把 Header、Footer、Main、Aside 都做成独立组件，并提供单独导出，既方便在模板中声明页面结构，也便于按块控制高度、宽度和样式。

**R**

这样布局既语义清晰，也便于组合复用，说明我在设计组件库时不只关心单个组件，还会考虑组件间协作体验。

## 4.5 Collapse / CollapseItem

### 组件定位

Collapse 是一个典型的父子联动组件，重点在状态共享和交互模式切换。

### 高频问题 1：Collapse 是怎么和 CollapseItem 通信的？

**S**

如果每个 CollapseItem 都自己维护开关状态，父组件就无法统一控制展开项。

**T**

我要让父组件统一维护激活项，子组件只负责触发和消费上下文。

**A**

我在 `Collapse.vue` 里通过 `provide` 提供 `activeNames` 和 `handleItemClick`，`CollapseItem.vue` 通过 `inject` 获取上下文，再用 `computed` 判断自己是否激活。点击标题时，如果不是禁用项，就调用父组件的切换方法。

**R**

这样父组件和子组件职责清晰，既支持外部 `v-model` 控制，也支持内部点击切换。

### 高频问题 2：手风琴模式和普通模式的区别是怎么处理的？

**S**

Collapse 既可能允许多项展开，也可能只允许一个展开。

**T**

我要在一套组件里同时兼容两种交互模式。

**A**

普通模式下，我维护的是一个数组，点击时做 `push/splice`；手风琴模式下则只保留一个激活值，如果当前点的是已展开项，就置空，否则替换成当前项。

**R**

这样 `accordion` 只是一个模式开关，不需要为不同场景维护两套实现，组件复用性更强。

## 4.6 Icon

### 组件定位

Icon 看起来是包装组件，但它承担了第三方图标库适配、属性透传和统一样式控制的职责。

### 高频问题 1：为什么不直接 everywhere 用 Font Awesome，而要包一层 Icon？

**S**

直接在每个业务组件里使用第三方图标组件，会让项目和第三方库强耦合。

**T**

我要把第三方图标库的接入细节收口到组件库内部。

**A**

我封装了 `Icon.vue` 作为统一入口，内部使用 `FontAwesomeIcon`，对外只暴露更符合组件库风格的 props，同时在 `main.ts` 和文档主题里统一注册图标库。

**R**

这样后续如果换图标库或增加统一样式能力，不需要改每个组件的实现，维护成本更低。

### 高频问题 2：为什么 `inheritAttrs: false` 和 `omit(props, ['color'])` 很关键？

**S**

Icon 组件既要把多数属性传给底层图标组件，又要保留一部分属性自己处理。

**T**

我要避免属性重复透传或样式控制失效。

**A**

我用 `inheritAttrs: false` 避免所有属性默认落到最外层，再通过 `omit` 把 `color` 从 props 中移除，因为 `color` 由组件自己转成 CSS 变量 `--color` 控制，其他属性再透传给 `FontAwesomeIcon`。

**R**

这样实现能证明我不是简单包一层，而是有意识地处理组件边界和属性分发。

## 4.7 Link

### 组件定位

Link 是对原生 `a` 标签的样式化和行为约束封装。

### 高频问题 1：Link 组件怎么处理 disabled？

**S**

原生 `a` 标签没有真正的 `disabled` 语义，如果处理不当，用户仍然可以跳转。

**T**

我要让 Link 在视觉和行为上都体现禁用状态。

**A**

我在组件里通过 `disabled || !href ? '' : href` 和 `target` 条件透传控制跳转能力，同时在点击事件里只有非禁用状态才触发 `emit('click', event)`，再配合类名做样式禁用。

**R**

这样 disabled 不只是样式变灰，而是行为层也被限制住了。

### 高频问题 2：Link 和 Button 的边界怎么划分？

**S**

很多设计系统里 Link 和 Button 很像，容易被混用。

**T**

我要说明两者在语义和场景上是不同的。

**A**

Button 更偏动作触发，比如提交、确认、删除；Link 更偏导航跳转和外链访问。实现上 Button 用原生 `button`，Link 用原生 `a`，两者保留了各自的语义能力。

**R**

这样在回答组件设计时，我能体现对“视觉相似但语义不同”的理解。

## 4.8 Tooltip

### 组件定位

Tooltip 是项目里非常关键的基础浮层组件，很多复杂组件都在复用它。

### 高频问题 1：Tooltip 为什么是这个项目里很重要的底层组件？

**S**

像 Dropdown、Select 这类组件都需要弹层定位和显示隐藏逻辑，如果每个组件都单独实现，代码会重复而且不稳定。

**T**

我要把浮层相关能力抽成一层基础设施。

**A**

我把 `Tooltip.vue` 设计成通用浮层组件，支持 `trigger`、`placement`、`manual`、`openDelay`、`closeDelay`、`click-outside` 和 `visible-change`，并通过 `defineExpose` 暴露 `show/hide` 方法给上层组件复用。

**R**

最终 Tooltip 成了整个交互组件体系里的基础能力层，Dropdown 和 Select 都是在它之上构建的。

### 高频问题 2：Tooltip 是怎么做定位的？

**S**

浮层组件难点之一是位置计算，如果纯手写，边界情况会很多。

**T**

我要让浮层在不同触发点和不同位置下都能稳定定位。

**A**

我使用了 `@popperjs/core` 的 `createPopper`，在 `isOpen` 变为 `true` 且节点就绪后创建实例，并通过 `placement` 和 `offset` modifier 控制弹层位置，同时在卸载时销毁实例避免内存泄漏。

**R**

这样我把“复杂几何计算”交给成熟库处理，把自己的关注点放在显示隐藏和交互逻辑上。

### 高频问题 3：Tooltip 为什么同时支持 hover、click 和 manual？

**S**

不同业务组件对浮层的控制方式不同，只支持单一触发方式会限制复用。

**T**

我要让 Tooltip 既能独立使用，也能被其他组件命令式控制。

**A**

我在组件内部根据 `trigger` 动态绑定事件；`hover` 时绑定 `mouseenter/mouseleave`，`click` 时绑定点击切换；如果 `manual` 为真，则不自动绑定触发事件，而是交给外部调用 `show/hide`。

**R**

这让 Tooltip 既适合作为普通提示组件，也适合作为 Dropdown、Select 这类复杂交互组件的底层能力。

### 高频问题 4：为什么要做 click outside？

**S**

浮层类组件常见问题是打开后无法自然关闭，尤其是点击触发时。

**T**

我要保证组件在用户点击外部区域时能正确收起。

**A**

我封装了 `useClickOutside` hook，在 Tooltip 容器挂载时监听文档点击，如果点击不在当前容器内，就根据触发模式决定是否关闭，并抛出 `click-outside` 事件给上层。

**R**

这样组件交互更符合用户预期，也方便 Select 这类依赖外部关闭信号的组件复用。

## 4.9 Dropdown

### 组件定位

Dropdown 是对 Tooltip 的一次业务封装，重点在菜单渲染和选择行为。

### 高频问题 1：Dropdown 是怎么复用 Tooltip 的？

**S**

如果 Dropdown 自己再写一套弹层逻辑，会和 Tooltip 重复。

**T**

我要让 Dropdown 只关心菜单语义，不重复造浮层轮子。

**A**

我直接在 `Dropdown.vue` 里把 `Tooltip` 作为外层容器，菜单列表放到 `content` 插槽中，再把 `trigger`、`placement`、`openDelay`、`closeDelay`、`manual` 等能力直接透传给 Tooltip。

**R**

这样 Dropdown 只负责菜单项渲染和选择事件，代码复用度更高。

### 高频问题 2：为什么要支持 `hideAfterClick` 和 `RenderVnode`？

**S**

菜单项并不总是纯文本，而且点击后是否自动收起也依赖交互场景。

**T**

我要让 Dropdown 的内容渲染和交互策略都更灵活。

**A**

我通过 `RenderVnode` 兼容字符串和 VNode 作为菜单 label，同时提供 `hideAfterClick` 控制点击菜单项后是否关闭弹层；若菜单项禁用，则不触发 `select`。

**R**

这样 Dropdown 不只是简单列表，而是具备了更接近业务可用的可配置性。

## 4.10 Input

### 组件定位

Input 是整个表单体系的基础组件，重点不只是输入框 UI，而是双向绑定、插槽能力和表单联动。

### 高频问题 1：Input 是怎么实现 `v-model` 的？

**S**

组件库里的 Input 需要兼容外部传值和内部输入，如果状态不同步，很容易出现体验问题。

**T**

我要保证 props、内部状态和事件输出三者一致。

**A**

我在组件内部维护 `inputValue`，用 `watch(() => props.modelValue)` 同步外部值变化；用户输入时更新内部值，并触发 `update:modelValue` 和 `input` 事件。

**R**

这样 Input 既是受控组件，也能保持内部交互流畅，符合 Vue 里组件化 `v-model` 的常见模式。

### 高频问题 2：Input 为什么要支持 prefix、suffix、prepend、append？

**S**

真实业务里的输入框经常不只是一个裸 input，还会有图标、前后缀、附加说明或操作按钮。

**T**

我要让 Input 具备足够强的组合能力，而不是被固定成单一形态。

**A**

我通过具名插槽支持 `prefix`、`suffix`、`prepend`、`append` 四类扩展区域，让输入框既能放图标，也能放复合结构，同时保留原生输入体验。

**R**

这样 Input 能够适配更多业务形态，不需要为不同样式再拆很多变体组件。

### 高频问题 3：Input 是怎么和 Form 联动校验的？

**S**

如果 Input 和 Form 完全解耦，表单就无法根据输入事件做规则校验。

**T**

我要让 Input 在保持通用性的同时，能接入表单校验上下文。

**A**

我通过 `inject(formItemContextKey)` 获取当前表单项上下文，在 `input`、`change`、`blur` 等事件里按触发时机调用 `formItemContext.validate(trigger)`，同时通过 `validateEvent` 控制是否启用这套联动。

**R**

这样 Input 既可以单独使用，也能在表单里自动参与校验，是一个比较典型的“独立组件 + 表单增强”的设计。

## 4.11 Rate

### 组件定位

Rate 是评分组件，重点是交互直观和状态更新简单清晰。

### 高频问题 1：Rate 的核心实现为什么要抽成 hook？

**S**

Rate 虽然不复杂，但如果逻辑都堆在组件里，会显得模板和状态耦合。

**T**

我要把评分状态管理抽成更独立的逻辑单元。

**A**

我用 `useRate` hook 去管理当前评分值和设置函数，组件层只负责渲染星星和触发点击事件，点击后再通过回调抛出 `changeRateNums`。

**R**

这样即使是简单组件，我也在练习“逻辑抽离”的工程习惯。

### 高频问题 2：Rate 当前实现还有什么可优化点？

**S**

面试官经常会通过“你自己怎么看这个组件”来判断你的复盘能力。

**T**

我要诚实说明当前实现完成了什么，也知道还能往哪里迭代。

**A**

目前它已经支持 `max`、`size`、选中颜色和未选中颜色，但事件名 `changeRateNums` 不够通用，后续可以统一成更标准的 `change`；另外还可以补 hover 预览、半星、键盘操作和无障碍属性。

**R**

这类回答能体现我不是把组件写完就结束，而是会主动做 API 和交互层的复盘。

## 4.12 Switch

### 组件定位

Switch 重点在于布尔/枚举值切换，而不是单纯样式变化。

### 高频问题 1：Switch 为什么要支持 `activeValue` 和 `inactiveValue`？

**S**

业务里开关的值不一定是 `true/false`，也可能是 `1/0`、`open/close` 这类枚举值。

**T**

我要让 Switch 组件适配更多数据模型。

**A**

我在类型里把 `modelValue`、`activeValue`、`inactiveValue` 都定义成 `string | number | boolean`，内部通过比较 `modelValue === activeValue` 来判断当前是否打开，再在切换时抛出新值。

**R**

这样 Switch 可以直接对接更多业务字段，不需要外部再额外做值转换。

### 高频问题 2：Switch 的文字和颜色为什么也做成可配置？

**S**

很多场景下开关不仅需要状态，还需要文字提示和业务语义色。

**T**

我要让它既能做通用开关，也能做更具业务表达力的组件。

**A**

组件支持 `activeText`、`inactiveText`、`activeColor`、`inactiveColor`、`inlinePrompt` 等配置，通过这些 props 决定开关两侧文案和当前配色。

**R**

最终 Switch 不只是一个开关壳子，而是能承载状态说明的交互组件。

## 4.13 Select

### 组件定位

Select 是项目里复杂度最高的输入类组件之一，核心难点在“输入框 + 弹层 + 选项状态 + 过滤逻辑”的组合。

### 高频问题 1：为什么 Select 不直接自己写弹层，而是复用 Tooltip？

**S**

Select 和 Tooltip、Dropdown 都有弹层逻辑，如果各写一套，维护成本高。

**T**

我要让 Select 聚焦在选择行为，而不是重复处理定位和弹层生命周期。

**A**

我让 Select 基于 `ElTooltip` 的 `manual` 模式实现弹层显示隐藏，再借助自定义 Popper modifier 做弹层和输入框同宽，点击外部时关闭。

**R**

这样 Select 的结构更清晰，也体现了我在复杂组件设计时优先考虑能力复用。

### 高频问题 2：Select 的过滤和远程搜索是怎么设计的？

**S**

下拉选择器常见需求不只是展示选项，还包括本地过滤和远程搜索。

**T**

我要在一个组件内同时兼容默认过滤、自定义过滤和远程搜索。

**A**

我把过滤逻辑统一收敛在 `generateFilterOptions` 里：有 `filterMethod` 时优先走自定义过滤；开启 `remote` 且存在 `remoteMethod` 时走异步远程搜索；否则走默认的 `label.includes()` 过滤。为了避免过于频繁触发，我还用 `debounce` 做了防抖。

**R**

这样 Select 既适合静态数据场景，也适合接口搜索场景，扩展性更强。

### 高频问题 3：Select 为什么要维护 `inputValue` 和 `selectOption` 两套状态？

**S**

如果只有一个状态，很容易在“显示文本”和“提交值”之间混淆。

**T**

我要区分“用户看到的内容”和“组件真正选中的值”。

**A**

我让 `inputValue` 负责输入框当前显示的文本，`selectOption` 负责真正选中的 option 对象。这样在过滤场景下，输入框可以暂时显示搜索词，而不破坏最终选中值。

**R**

这种状态拆分能让交互更稳定，也说明我在设计复杂组件时会主动拆开显示态和业务态。

### 高频问题 4：你怎么看 Select 当前实现还能优化什么？

**S**

面试官问这类问题，本质上是在看你是否具备继续维护和迭代组件的能力。

**T**

我要说明当前实现的完成度，同时明确下一步优化方向。

**A**

目前它已经覆盖单选、清空、禁用、过滤、远程搜索和自定义渲染，但还可以继续补多选、虚拟滚动、键盘导航、远程请求取消、加载中 UI 和空状态文案。

**R**

这样的回答能体现我有产品化意识，不会把组件理解成“一次性作业”。

## 4.14 Form / FormItem

### 组件定位

Form 是整个项目里最能体现组件通信和工程抽象能力的组件之一。

### 高频问题 1：为什么 Form 需要维护 `fields` 注册表？

**S**

表单的难点不在于渲染容器，而在于如何统一管理多个字段实例。

**T**

我要让 Form 能在外部调用 `validate`、`resetFields`、`clearValidate` 时，准确调度所有 FormItem。

**A**

我在 `Form.vue` 中维护了一个 `fields` 数组，`FormItem` 在 `onMounted` 时通过 `addField` 注册，在 `onUnmounted` 时通过 `removeField` 移除。这样表单就掌握了当前参与校验的字段集合。

**R**

最终 Form 可以做整表级调度，而不是被动依赖每个字段各自处理。

### 高频问题 2：FormItem 的校验为什么交给 async-validator？

**S**

校验规则一旦变复杂，自己手写规则引擎既费时也不稳定。

**T**

我要在项目里采用一个成熟、可复用的规则校验方案。

**A**

我让 `FormItem` 根据 `prop` 去拿 `rules`，再根据 `trigger` 过滤出当前事件应该执行的规则，最后交给 `async-validator` 去校验，并把返回的错误信息同步到 `validateStatus` 中。

**R**

这样既降低了手写规则逻辑的复杂度，也能把表单能力扩展到更接近真实业务的程度。

### 高频问题 3：Form、FormItem、Input 三者的职责是怎么划分的？

**S**

如果职责划分不清，表单组件很容易越写越乱。

**T**

我要把“调度”“单字段校验”“输入行为”这三类职责拆开。

**A**

Form 负责字段注册和整表调度，FormItem 负责当前字段的规则匹配、错误状态和结果展示，Input 只负责输入行为和触发校验时机。三者之间通过 `provide/inject` 协作，而不是互相直接耦合。

**R**

这使整个表单体系更像一套可维护的组件系统，而不是几段拼起来的功能代码。

### 高频问题 4：如果面试官问“你觉得 Form 最体现你什么能力”，怎么答？

**S**

Form 是组件库里典型的高复杂度组件，能很集中地体现工程抽象能力。

**T**

我要让面试官看到的不只是我会写一个表单页面，而是我能抽成通用表单系统。

**A**

我会重点讲字段注册、规则过滤、上下文通信、单字段和整表校验这几件事，以及我是怎么把这些能力拆到 Form、FormItem、Input 三层的。

**R**

这通常能很好体现我的组件抽象、状态建模和代码组织能力。

## 4.15 Dialog

### 组件定位

Dialog 是一个带生命周期、关闭策略和交互增强的模态组件。

### 高频问题 1：Dialog 的显示和销毁为什么要分成 `visible` 和 `rendered`？

**S**

如果只用一个状态控制显示，很难兼顾过渡动画和销毁时机。

**T**

我要让 Dialog 既能有离场动画，又能支持 `destroyOnClose`。

**A**

我用 `visible` 控制当前是否显示，用 `rendered` 控制 DOM 是否仍然保留。打开时先确保 `rendered=true`，关闭时先隐藏，动画结束后如果 `destroyOnClose` 为真，再把 `rendered` 置为 `false`。

**R**

这样动画和销毁两件事不会互相打架，组件行为也更完整。

### 高频问题 2：Dialog 的拖拽是怎么实现的？

**S**

拖拽弹窗的本质是记录鼠标位移并映射到弹窗位置。

**T**

我要在不影响弹窗内容交互的前提下，只允许拖头部区域移动。

**A**

我在 header 的 `mousedown` 里记录起始点，再在全局 `mousemove` 中持续更新 `dragState.x/y`，最后把位移映射成 `transform: translate(x, y)`。鼠标抬起后则停止拖拽。

**R**

这样实现简单直接，也比较容易测试和复盘。

### 高频问题 3：为什么要做锁滚动和 ESC 关闭？

**S**

模态框打开时，如果页面还能随意滚动或者 ESC 不生效，会影响用户体验。

**T**

我要让 Dialog 具备常见交互规范。

**A**

打开时我把 `document.body.style.overflow` 改成 `hidden` 并保存旧值，关闭后恢复；同时在文档级监听 `keydown`，当检测到 `Escape` 且配置允许时触发关闭。

**R**

这样 Dialog 更符合真实产品里的模态交互预期。

### 高频问题 4：`beforeClose` 的设计意义是什么？

**S**

很多业务场景下，关闭弹窗前需要二次确认或执行异步逻辑。

**T**

我要给业务层一个拦截关闭流程的机会。

**A**

我提供了 `beforeClose` 回调，并把 `done` 函数传给它。业务层可以在回调里决定是否继续关闭，而组件本身只负责执行这个约定。

**R**

这样 Dialog 的控制力就不仅停留在 UI 层，而是能和业务逻辑很好衔接。

## 4.16 DatePicker

### 组件定位

当前 DatePicker 更接近一个轻量级日历选择器，重点在日历表格生成和月份切换。

### 高频问题 1：DatePicker 的 42 宫格是怎么生成的？

**S**

日历组件不是简单列一个月的日期，而是需要补齐前月和后月，形成完整周视图。

**T**

我要生成固定 7x6 的日历网格，保证布局稳定。

**A**

我把日历计算逻辑抽到 `calendar.ts`，先拿当前月天数和当月 1 号是周几，再补齐上个月尾部和下个月头部日期，最终生成 42 个单元格。

**R**

这样页面渲染时布局稳定，测试里也可以直接验证网格数量。

### 高频问题 2：DatePicker 当前实现还有什么不足？

**S**

这个组件属于阶段性版本，功能上还没有做到特别完整。

**T**

我要诚实说明现状，同时给出合理迭代方向。

**A**

当前版本主要支持月份切换和当天高亮，但还可以继续补单击选中日期、面板收起、格式化输出、范围选择、键盘导航等能力。另外闰年判断逻辑后续也值得再仔细校验和完善。

**R**

这样回答既真实，也能体现我对日期类组件复杂度的认识。

## 4.17 Message

### 组件定位

Message 是一个典型的函数式全局反馈组件，难点在动态创建、堆叠管理和自动销毁。

### 高频问题 1：为什么 Message 适合做成函数式组件？

**S**

消息提示通常是“随时弹出一下”，并不适合要求业务页面先写好模板节点。

**T**

我要让业务层能像 `Message.success()` 这样直接调用。

**A**

我通过 `h + render` 动态创建 `Message.vue` 的 vnode，再把它挂到新建容器上，同时支持字符串和 VNode 作为消息内容。

**R**

这样消息提示的调用方式更接近成熟组件库的使用习惯。

### 高频问题 2：Message 是怎么做堆叠的？

**S**

全局消息经常会连续触发，如果每条消息都固定在同一个位置，就会重叠。

**T**

我要让多条消息按顺序向下堆叠展示。

**A**

我维护了一个 `instances` 实例数组，每次创建新消息时通过 `getLastBottomOffset` 获取前一条消息的底部位置，再计算当前消息的 `topOffset` 和 `bottomOffset`。

**R**

这样多条消息会自然错开，而不是互相覆盖。

### 高频问题 3：Message 为什么要单独做 `useZIndex`？

**S**

弹层和全局反馈组件如果没有统一层级策略，后面很容易互相遮挡。

**T**

我要给所有动态反馈类组件一个可复用的层级递增机制。

**A**

我封装了 `useZIndex` hook，内部维护一个全局递增值，每次创建新实例时拿 `nextZIndex()` 作为当前消息的层级。

**R**

这样层级控制就不会散落在各个组件里，更容易统一管理。

### 高频问题 4：Message 的关闭流程是怎么设计的？

**S**

消息提示既可能自动消失，也可能手动关闭，还要兼顾离场动画。

**T**

我要让关闭动作、动画和最终销毁串成完整流程。

**A**

组件挂载后启动定时器，到时把 `visible` 设为 `false`；手动点击关闭也是同样逻辑。真正的销毁在离场动画结束后，通过传入的 `onClose` 回调把实例从池中移除并 `render(null, container)` 卸载。

**R**

这套设计让我对命令式组件的完整生命周期有了比较深入的理解。

## 4.18 MessageBox

### 组件定位

MessageBox 是函数式确认弹窗，核心在 Promise 化返回结果。

### 高频问题 1：为什么 MessageBox 要返回 Promise？

**S**

确认弹窗最典型的使用方式就是“等用户点确认或取消之后再继续执行逻辑”。

**T**

我要让它的使用方式贴近业务控制流。

**A**

我把 `MessageBox(options)` 设计成返回 `Promise<ActionType>`，用户点击 `confirm`、`cancel` 或关闭时，内部通过监听 `vm.state` 的变化来 `resolve` 对应动作。

**R**

这样业务层可以直接 `await MessageBox.confirm(...)`，代码体验更自然。

### 高频问题 2：MessageBox 和 Dialog 的关系是什么？

**S**

面试官经常会问两个弹窗组件的边界是否清楚。

**T**

我要区分“通用容器弹窗”和“预置流程弹窗”。

**A**

Dialog 更像一个声明式容器组件，适合承载任意 slot 内容；MessageBox 是函数式预置弹窗，主要服务于消息提示和确认流转，强调的是快速调用和结果返回。

**R**

这样可以体现我对组件边界和交互模型的清晰理解。

### 高频问题 3：为什么用 `createApp` 而不是 `h + render`？

**S**

函数式组件有多种动态创建方式，不同场景适合不同方案。

**T**

我要选择更适合确认弹窗这种“独立小应用实例”的实现方式。

**A**

MessageBox 里我用了 `createApp(MessageBoxComponent, options)`，因为它本身更像一次独立挂载，内部状态和生命周期也相对完整，再通过 `mount` 和 `watch(vm.state)` 去控制最终结果。

**R**

这说明我不是机械套同一种做法，而是会根据组件类型选择更合适的动态挂载方式。

## 5.1 Vue 3

### 问题 1：为什么这个项目统一用 Vue 3 的组合式 API？

**S**

组件库里有很多跨组件复用逻辑，比如表单校验、z-index、点击外部关闭，这些逻辑如果都写在 Options API 里，复用体验一般。

**T**

我要选择一种更适合抽离逻辑和组织复杂组件的开发方式。

**A**

我主要用 `<script setup>`、`ref`、`computed`、`watch`、`provide/inject`、`defineExpose` 等组合式 API，让状态、行为和对外暴露更集中。像 `useZIndex`、`useClickOutside`、`useRate` 都是这种模式下比较自然的抽离结果。

**R**

这套写法让我在复杂组件里能更清晰地组织逻辑，也更适合做组件库项目。

### 问题 2：这个项目里哪些地方最能体现 Vue 3 的组件通信能力？

**S**

组件通信是面试官很喜欢深挖的点。

**T**

我要用真实代码举出不同层级的通信方式。

**A**

父子同步场景主要用 `props + emits + v-model`，比如 Input、Switch、Dialog；跨层通信主要用 `provide/inject`，比如 Collapse 和 Form 体系；命令式能力暴露主要用 `defineExpose`，比如 Tooltip、Dialog、Button、Message。

**R**

这样我可以告诉面试官，我不是只会一种通信方式，而是会按问题类型选方案。

### 问题 3：为什么 Form 里用 `provide/inject`，而不是层层传 props？

**S**

Form 场景里有多级嵌套组件，层层透传会让模板越来越重。

**T**

我要在不增加外部使用复杂度的前提下，把表单能力传到最内层。

**A**

我让 Form 提供字段注册和校验能力，FormItem 消费后再进一步向 Input 提供字段级校验上下文，这样就形成了两层上下文传递，不需要外层模板知道中间层的细节。

**R**

这说明我理解 Vue 3 的 `provide/inject` 最适合用来做“祖先能力下发”，尤其是表单这种天然上下文型组件。

### 问题 4：`watch` 和 `computed` 在这个项目里你是怎么区分使用场景的？

**S**

面试官经常通过这个问题判断你对响应式系统的理解是否扎实。

**T**

我要明确说明“派生值”和“副作用”的边界。

**A**

派生值我用 `computed`，比如 Button 的类名、Dialog 的样式、Select 的 placeholder；副作用我用 `watch`，比如监听 `modelValue` 同步内部状态、监听 `isOpen` 创建 Popper、监听表单值变化触发流程。

**R**

这样的回答能说明我不是机械地用 API，而是清楚每个 API 的职责边界。

### 问题 5：为什么项目里会同时出现 `h/render/createApp`？

**S**

这几个 API 都和 Vue 的命令式渲染有关，很多人只停留在概念层。

**T**

我要结合真实项目说明它们分别适合什么场景。

**A**

`h` 用来手动创建 vnode，`render` 适合像 Message 这种直接控制 vnode 挂载和卸载的场景；`createApp` 更适合 MessageBox 这种完整组件实例化并独立挂载的场景。

**R**

这能体现我不仅会写模板式 Vue，也理解底层一点的渲染和实例创建方式。

### 问题 6：为什么大量使用 `script setup`？

**S**

组件库里组件很多，样板代码越多，维护成本越高。

**T**

我要用更简洁的语法降低组件模板噪音。

**A**

`script setup` 省掉了大量样板式 `setup()` 返回值，同时搭配 `defineProps`、`defineEmits`、`defineExpose`、`defineOptions`，可以把组件定义集中到一个区域。

**R**

这样开发效率更高，代码也更贴近“组件声明”本身。

**R**

一套工具同时覆盖开发和打包，工程复杂度更低。

### 问题 2：这个项目的库模式构建是怎么做的？

**S**

组件库和普通单页应用最大的差异之一，就是它需要面向“被别人安装使用”去构建。

**T**

我要输出适合发布的构建产物，而不是只生成一个页面包。

**A**

我在 `vite.config.ts` 里根据 `mode === 'lib'` 开启 `build.lib`，入口指向 `src/index.ts`，输出 `tiny-element.js` 和 `tiny-element.umd.js`，格式为 `es` 和 `umd`。

**R**

这样项目对外既可以走现代 ESM，也能兼容需要 UMD 的环境。

### 问题 3：为什么要把 Vue 设为 external？

**S**

组件库打包时如果把 Vue 一起打进去，很容易造成宿主项目重复依赖和体积膨胀。

**T**

我要让库产物更轻，并避免和宿主项目的 Vue 实例冲突。

**A**

我在 `rollupOptions.external` 里把 `vue` 排除出去，同时在 `output.globals` 里为 UMD 模式声明全局变量名 `Vue`。

**R**

这样库产物更符合组件库发布习惯，也体现了我对构建边界的理解。

### 问题 4：类型声明是怎么输出的？

**S**

组件库如果只有运行时代码，没有类型声明，TypeScript 用户体验会很差。

**T**

我要让外部使用者在引入组件时也能拿到类型提示。

**A**

我通过 `vue-tsc -p tsconfig.build.json` 生成 `dist/types` 下的声明文件，并在 `package.json` 里通过 `types` 和 `exports` 指向声明入口。

**R**

这样库对 TS 用户更友好，也更接近成熟 npm 包的交付方式。

### 问题 5：Vite 配置里还有哪些关键点？

**S**

面试官有时会进一步看你是否只知道“启动快”，还是知道项目里具体怎么配置。

**T**

我要结合项目真实配置回答。

**A**

项目里配了 `@vitejs/plugin-vue` 和 `vite-plugin-eslint`，还配置了 `@` 和 `@components` 路径别名；文档站里则额外通过 Vite 配置了 `/api/rag` 到本地 Node 服务的代理。

**R**

这说明我对 Vite 的理解不只停留在使用命令，而是能落到具体工程配置。

## 5.3 TypeScript

### 问题 1：TypeScript 在这个项目里最大的价值是什么？

**S**

组件库一旦组件数量变多，没有类型约束，props 和事件会越来越难维护。

**T**

我要让组件接口更稳定、更可维护。

**A**

我在项目里用 TS 约束 props、emits、组件实例、表单规则、上下文注入 key、消息实例类型等，比如 Form 的 `FormContext`、Tooltip 的 `TooltipInstance`、Message 的 `MessageContext`。

**R**

这样项目后续重构和扩展组件时更有安全感，也更容易被 IDE 正确提示。

### 问题 2：为什么有些组件用 `interface`，有些地方用 `PropType`？

**S**

Vue 组件开发里同时存在运行时 props 校验和编译时类型系统，两者要配合使用。

**T**

我要兼顾运行时和类型层的约束。

**A**

像 Button 这类组件，我用 `buttonProps` 配合 `PropType` 去做运行时 props 定义；而像 Dialog、Select、Tooltip 这类更多是纯类型描述时，我会直接用 `interface` 配合 `defineProps<T>()`。

**R**

这说明我理解 Vue + TS 里“运行时声明”和“类型声明”是两个维度，不会混为一谈。

### 问题 3：为什么 `InjectionKey` 很重要？

**S**

`provide/inject` 如果只是字符串 key，很容易在复杂项目里丢失类型信息。

**T**

我要让上下文通信也保持类型安全。

**A**

我在 Form 和 Collapse 里都用了 `InjectionKey<T>`，比如 `formContextKey`、`formItemContextKey`、`collapseContextKey`，这样注入时能直接拿到精确类型。

**R**

这让跨层通信不只是“能传”，而是“传得安全、传得可维护”。

### 问题 4：如果面试官问“TypeScript 给你带来了什么真实收益”，怎么答？

**S**

很多人会把 TypeScript 说成“只是有提示”，但面试里最好说得更具体。

**T**

我要把收益落到真实组件开发场景里。

**A**

我会说收益主要有三点：第一，组件 API 更清晰，调用时不容易传错；第二，复杂上下文对象比如 Form 和 Message 的实例管理更稳定；第三，构建时可以顺手输出声明文件，提升组件库的交付质量。

**R**

这类回答更像真实工程经验，而不是背概念。

## 5.4 Pinia

### 问题 1：这个项目为什么没有用 Pinia？

**S**

面试官如果看到你了解 Pinia，通常还会继续问“那你为什么没用”。

**T**

我要先诚实交代，再说明技术判断依据。

**A**

当前项目是组件库，不是复杂业务系统，状态主要集中在组件内部和父子通信，比如 Input、Dialog、Select、Form 这类都属于局部状态或上下文状态，暂时没有全局共享状态场景，所以没有引入 Pinia。

**R**

这说明我不是“看到热门库就强行上”，而是会根据项目边界做选型。

### 问题 2：如果后续要在这个项目里接入 Pinia，你会用在什么地方？

**S**

虽然当前项目没接入 Pinia，但如果项目继续演化，可能会出现全局状态需求。

**T**

我要说明自己知道 Pinia 应该放在什么场景下，而不是泛泛而谈。

**A**

如果后续文档站要加入全局主题、AI 聊天历史持久化、用户偏好设置、组件搜索历史，或者 demo 站要管理跨页面共享状态，我会考虑引入 Pinia，把这些“跨页面、跨组件、非局部”的状态集中管理。

**R**

这样我的回答既诚实，又能体现我知道 Pinia 的适用边界。

### 问题 3：Pinia 和 Vuex 最大的区别你怎么答？

**S**

这是状态管理面试里的高频基础题。

**T**

我要用实用角度说明差异，而不是只背历史。

**A**

我会说 Pinia 更轻量、API 更贴近组合式 API，去掉了 Vuex 里比较重的 mutation 概念，状态修改通常直接在 action 里做；同时 TypeScript 支持更自然，模块化体验也更好。

**R**

这个回答能体现我关注的是开发体验、类型支持和使用成本。

### 问题 4：Pinia 的核心概念有哪些？

**S**

如果面试官继续深挖，通常会看你是否真的理解 store 的组成。

**T**

我要把核心概念回答完整。

**A**

Pinia 的核心是 `defineStore`，每个 store 里通常包含 `state`、`getters`、`actions`。组件中使用时会创建 store 实例，如果需要保持响应式解构，一般会配合 `storeToRefs` 使用。

**R**

这说明我对 Pinia 的使用模型是清楚的，即使当前项目没落地，也能接住原理追问。

### 问题 5：如果面试官问“什么时候不用 Pinia”，你怎么答？

**S**

很多人会把全局状态管理当成默认配置，但这其实不总是对的。

**T**

我要说明自己知道什么时候应该保持简单。

**A**

如果状态只在单个组件内部使用，或者只是明确的父子层级传递，我会优先用组件局部状态、props/emits、provide/inject；只有当状态跨页面、跨模块共享，且更新逻辑复杂时，我才会考虑 Pinia。

**R**

这会让面试官觉得我在技术选型上是有判断力的。

## 5.5 Vitest / 测试

项目本身基于 Vite，测试工具如果能和构建体系保持一致，开发体验会更好。

面试官经常会问“你说有测试，那到底测了什么”。

**T**

我要回答到真实用例层，而不是空泛说“做了单测”。

目前主要覆盖了三类：基础渲染和事件触发，比如 Button、Alert、Link、Icon；交互型组件，比如 Collapse、Dialog、Dropdown、Select、Switch、Tooltip；表单联动，比如 Input 和 Form 校验流程。

## 5.7 async-validator

### 问题 1：为什么表单校验选择 async-validator？

**S**

表单规则一旦变多，自己写校验逻辑会越来越杂。

**T**

我要找一个成熟、可配置、和表单组件契合的规则校验方案。

**A**

我选择 `async-validator`，因为它对表单字段规则建模清晰，也支持错误信息结构化返回，比较适合在 FormItem 内部做字段级校验。

**R**

这样能把重点放在表单框架设计上，而不是重复造规则引擎。

### 问题 2：你是怎么把 trigger 和 async-validator 结合起来的？

**S**

不同规则可能在不同交互时机触发，比如 blur、change、input。

**T**

我要保证规则执行时机符合表单交互预期。

**A**

我在 `FormItem` 里先从当前字段规则中筛出和当前 trigger 匹配的规则，再把 `trigger` 字段去掉后交给 `async-validator` 去执行。

**R**

这样表单校验就不只是“有规则”，而是“在正确时机执行正确规则”。

## 5.8 Popper / 浮层定位

**A**

**A**

我会说自己理解浮层定位至少要处理参考点、偏移、滚动容器和边界检测，但在工程实践里，我更倾向于把成熟问题交给 Popper，自己重点做好交互层封装和复用。

## 5.10 RAG / Node / Qdrant

## 7. 最后冲刺建议

- 简单组件不要只答“这个很简单”，要答出“为什么要单独抽成组件”。
- 复杂组件尽量围绕“职责拆分、状态流转、复用关系、可扩展性”来回答。
- 技术栈深挖时，先讲这个项目里真实怎么用，再讲原理和扩展。
- 问到 Pinia 时一定先说“这个项目没用”，再展开，不要硬编。
- 如果面试官连续追问，优先从 Form、Select、Tooltip、Dialog、Message、RAG 这几个部分展开。
