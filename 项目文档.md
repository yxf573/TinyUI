1. 项目难点
我觉得最核心的两个难点是 Message 的函数式调用 和 Form 表单校验体系。

(1) Message 组件的难点在于它不是普通模板组件，而是要通过 Message.success() 这种函数方式调用：Message.success('保存成功')
· 组件实例在哪里创建？多个Message同时出现怎么进行管理？
· 然后，我就把 Message 拆成两层：一层是 Message.vue，只负责 UI 展示、定时关闭、偏移量计算；另一层是 method.ts，作为服务层，负责动态创建实例和管理全部实例。
· 在创建一条消息的时候，用 h()函数 创建 VNode，用 render() 渲染到动态创建的 div 容器里，然后挂载到 document.body。同时维护一个实例数组，每创建一条消息就放进去，每条消息都会创建唯一id。
· 多个Message堆叠的问题，通过每个实例都会暴露一个 底部位移 的字段来解决的。后一条 Message 会找到前一条 Message，读取它暴露出来的 bottomOffset，然后计算自己的 topOffset。这样消息就会一条一条往下排，而不是重叠在一起。
· 关闭的时候，我没有直接删除 DOM，而是先把 visible 改成 false，让离场动画<Transition>执行。动画结束后，Message.vue 通过 onClose 回调通知服务层，服务层再执行 render(null, container) 卸载组件，并把实例从 instances 中移除。这样既保证了动画效果，也避免了内存泄漏。

(2) Form 的难点在于它是跨组件协作的。
类似于下面的使用：
<ElForm :model="model" :rules="rules">
  <ElFormItem prop="name">
    <ElInput v-model="model.name" />
  </ElFormItem>
</ElForm>
· 传值选择；Form、FormItem、Input 三者之间需要通信，Input输入时触发FormItem校验，Form提交时又需要统一校验所有FormItem。如果使用props一层层传就很麻烦。
· 我就是使用provide/inject实现上下文通信。Form通过provide向下提供model, rules, addField, removeField, validate, resetFields方法。FormItem注入Form上下文，根据自己的prop找到对应的字段和校验规则。Input再注入FormItem上下文，再input, change, blur时调用FormItem的validate(trigger)方法。
· Form本身维护一个fields数组，FormItem挂载时就注册进去，卸载时移除，这样Form调用validate时就可以遍历所有FormItem，统一执行校验。

2. Message组件
Message 组件我是按服务型组件设计的。外部通过 Message.success() 这种函数式 API 调用，内部使用 h() 创建 VNode，再用 render() 挂载到动态创建的 DOM 容器中，并追加到 document.body。每个实例都会生成唯一 id，并保存到 instances 实例池里，用于关闭、销毁和层叠计算。

层叠计算是通过 bottomOffset 实现的。每条 Message 挂载后会计算自己的高度，然后得到 bottomOffset = topOffset + height，并通过 defineExpose 暴露出去。后一条 Message 会根据自己的 id 找到前一条实例，读取前一条的 bottomOffset，再加上默认 offset，作为自己的 topOffset。这样多条消息就能依次向下排列，关闭其中一条后，后面的消息也会重新计算位置并上移。

message.success()：就会转化为Message({..., type: 'success'})，通过type定制消息样式类型。
    设计了一个创建函数createMessage，将success作为type类型传入，接收用户传入的字符串、VNode 或配置对象，然后把参数统一转换成 MessageOptions，并自动补上 type: 'success'，最后调用基础的 message() 方法创建实例。

(3) 为什么 Message.vue 要用 defineExpose？
    因为 Message 的服务层需要控制和读取组件内部状态，比如计算堆叠位置时，服务层需要读取前一条 Message 的 bottomOffset。

(4) 关闭 Message 时，为什么不直接 remove DOM？
    如果直接消失，是不美观的，所以先让它隐藏，并展示离场动画，离场动画展示完后，就销毁组件
    <Transition name="fade-up" @after-leave="destroyComponent" @enter="updateHeight" >
        <!-- 消息容器 -->
    </Transition>
    /* 过渡动画 */
    .el-message.fade-up-enter-from,
    .el-message.fade-up-leave-to {
        opacity: 0;
        transform: translate(-50%, -100%);
    }

(5) 当前一条 Message 关闭后，后面的 Message 会不会自动上移？
    会。因为实例数组 instances 是响应式的，当前一条被 splice 删除后，后面 Message 计算 lastOffset 时，会重新根据实例数组找到自己的前一个实例。
    它的 topOffset 和 bottomOffset 是 computed，依赖实例数组和前一条的暴露状态，所以前面的消息销毁后，后面的消息位置会重新计算并上移。

(6) 自动关闭如何实现
    Message.vue 内部维护一个 timer。
    挂载后设置 visible = true，然后调用 startTimer()。如果 duration 不为 0，就开启定时器，到时间后把 visible 改为 false，触发离场动画。
    如果 duration 是 0，则表示不自动关闭。
    
    通过定时器控制 visible 状态实现的。组件挂载后设置 visible = true 并启动 setTimeout，到达 duration 时间后把 visible 改为 false。由于模板外层包了 Transition，所以 visible 变 false 后会先播放离场动画，动画结束后触发 after-leave，再调用 onClose 执行真正的卸载和 DOM 移除。

(7) z-index 是怎么处理的？
    一个函数统一管理z-index，Message 每创建一个实例时会调用 useZIndex().nextZIndex()，生成递增层级。这样后创建的 Message 层级更高，可以避免多个浮层或消息之间互相遮挡。

(8) Message 有什么可以优化的地方？
    1 可以支持最大消息数量，超过数量时自动关闭最早的消息。
    2 目前实现的Message消息是固定在顶部中间进行显示的，可以补充支持不同位置的显示选择，比如 top、bottom、top-right，不只固定顶部。

4. Form组件
(1) Form组件整体如何设计
    · Form 是表单控制中心，负责接收 model 和 rules，维护所有 FormItem 实例，并提供 validate、resetFields、clearValidate 等方法。
    · FormItem 是字段控制层，负责根据自己的 prop 找到字段值和校验规则，执行单字段校验，并维护错误状态。
    · Input 是具体输入组件，只负责值更新和触发校验，比如 input、change、blur 时通知 FormItem。

(2) 为什么不用 props 一层层传，而用 provide/inject？
    因为 Form、FormItem、Input 在使用时并不一定是直接父子关系，比如Form表单在提交的时候会对每个字段都进行验证，这个时候就是要具体的input进行判断，所以使用provide/inject，这样Form 不需要直接操作 Input，Input 也不需要知道整个 Form，只需要调用当前 FormItem 提供的 validate 方法。

(3) Form provide 了什么内容？
    表单上下文
    model // 表单数据
    rules // rules校验规则
    下面是表单能力
    addField
    removeField
    validate
    resetFields
    clearValidate

(4) FormItem provide 了什么内容？
    prop
    validate
    resetField
    clearValidate
    $el

(5) Form 是怎么知道有哪些 FormItem 的？
    Form 内部维护了一个 fields 数组。FormItem在挂载的时候，就把自己加入到这个数据组，在卸载时删除数组中的自己，这样 Form 就能知道当前表单里有哪些字段，调用 validate() 时就可以遍历所有字段。

(6) 为什么只有 FormItem 有 prop 才注册？
    只有设置了 prop，才表示它和 model 中某个字段绑定，需要参与校验、重置和清空校验。

(7) FormItem 是怎么找到当前字段值的？FormItem 是怎么找到校验规则的？
    FormItem 会从 Form 注入的 model 中，根据自己的 prop 取值。
    FormItem 会从 Form 注入的 rules 中，根据自己的 prop 找规则。
    例如：
    {
        name: [
            { required: true, message: '请输入姓名', trigger: 'blur' }
        ]
    }

(8) 为什么要引入 async-validator？
    async-validator 是成熟的校验库，项目里 FormItem 只负责取值、取规则、触发校验和展示状态，把具体规则判断交给 async-validator。

(9) 单字段校验流程是什么？
    Input 触发 blur/input/change
            ↓
    调用 FormItem.validate(trigger)
            ↓
    FormItem 根据 trigger 过滤 rules
            ↓
    用 async-validator 校验当前字段值
            ↓
    成功：validateStatus.state = success
    失败：validateStatus.state = error，并保存 errorMsg

(10) Form 的 validate 是怎么实现的？
    Form 的 validate() 会遍历 fields 数组，逐个调用每个 FormItem 的 validate('')

(11) 为什么要用 InjectionKey？
    InjectionKey<FormContext> 是 Vue 提供的类型工具，可以让 provide/inject 有 TypeScript 类型提示。

(12) 如果 FormItem 嵌套很深，inject 还能拿到吗？
    可以。provide/inject 是跨层级通信，只要 FormItem 是 Form 的后代，就能注入到 Form 提供的上下文，不要求是直接子组件。

(13) 这个 Form 实现有什么可以优化的地方？
    1 支持嵌套字段路径：目前只支持一级字段，
        prop="name"，prop="user.name"
        封装 getPropByPath 工具函数 —— 从对象里面获取到相应的值
    2 支持 trigger 为数组：可以让同一个校验规则，复用在不同的触发事件上
    3 异步校验结果乱序：有的字段可能需要远程校验，比如用户名是否重复，用户输入很快的时候需要在最后一个输入的校验结果为准，但是异步请求返回顺序不确定，
    给每次校验加一个递增的校验序号。每次执行校验时，先生成当前这次校验的 id。等异步校验返回后，再判断这个 id 是否还是最新的。如果不是最新的，说明这次结果已经过期，就直接丢弃，不再更新表单状态。（只接受最后一个结果）

(14) 异步校验async-validator的底层实现原理：
    把所有规则统一包装成异步任务，然后用 Promise 汇总结果。
    传给async-validator
    {
        username: [
                    { required: true, message: '请输入用户名' },
                    {
                        validator(rule, value) {
                            return checkUsername(value)
                    }
                }
            ]
    }
    它先把用户传入的 rules 规范化成字段到规则数组的映射，然后在 validate(source) 时遍历每个字段和每条规则，根据规则里的 type、required、pattern、validator 等信息找到对应的校验函数执行。每个校验函数会把错误通过 callback 或 Promise 返回，async-validator 再统一收集错误，最后如果没有错误就 resolve，有错误就 reject，并返回 errors 和按字段分组的 fields。

Button 组件为什么用动态 class，而不是直接写行内样式？
    Button 的状态很多，比如 primary、success、danger、disabled、loading、plain、round 等。
    如果用行内样式，逻辑和表现会混在一起，不利于维护。
    项目里通过 计算属性computedClass 根据 props 生成 el-button--primary、is-loading 这类类名，把状态逻辑留在组件。

Button 的 loading 状态做了什么？
    loading 为 true 时，按钮会显示 spinner 图标，同时原生 button 的 disabled 会变成 true。这样不仅视觉上告诉用户正在加载，也能防止重复点击。这个细节比单纯显示图标更完整，因为组件库要考虑交互状态的一致性。

Message 为什么要做成函数式调用？
    Message 通常不是写在模板里的静态组件，而是业务代码里临时触发的反馈，比如 Message.success('保存成功')。所以项目里用函数式方式封装：调用 message() 后，内部通过 h() 创建 VNode，用 render() 渲染到动态创建的 DOM 容器，再挂到 document.body 上。这样用户不需要在页面里手动写 <ElMessage />。

Message 动态销毁是怎么实现的？
    创建 Message 时会生成唯一 id，并把实例放入 instances 数组。关闭时不是马上删除 DOM，而是先把暴露出来的 visible 改成 false，让离场动画执行；动画结束后触发 after-leave，再调用 destroy，用 render(null, container) 卸载 Vue 实例，并从实例池中移除。

Message 多条消息堆叠时如何避免重叠？
    每个 Message 会计算自己的 topOffset。它会通过当前 id 找到自己在 instances 中的位置，然后拿前一个 Message 暴露出来的 bottomOffset，当前消息的顶部位置就是 offset + lastBottomOffset。Message 进入动画时会测量自身高度，得到 bottomOffset = topOffset + height，后面的消息就能基于它继续向下排列。

Message 为什么用 shallowReactive 存实例数组？
    实例池主要关心数组增删，以及根据顺序计算偏移量，不需要深度追踪 VNode、组件实例、props 里所有字段。如果用深响应式，反而会带来不必要的代理成本。shallowReactive 对这种“管理实例集合”的场景更合适。

Form 和 FormItem 为什么要用 provide/inject？
    Form、FormItem、Input 不是简单父子关系，中间可能有插槽和多层结构。如果靠 props 层层传递，组件使用会很不灵活。项目中 Form 通过 provide 提供 model、rules、addField、validate、resetFields 等上下文；FormItem 注入 Form 上下文；Input 再注入 FormItem 上下文来触发校验。这就是组件库常见的“上下文通信”模式。

Form 校验流程是怎样的？
    FormItem 根据自己的 prop 从 Form 的 model 中取当前字段值，从 rules 中取当前字段规则。触发校验时，会按 trigger 过滤规则，然后用 async-validator 创建校验器。校验成功就把状态改为 success，失败就把状态改为 error 并保存错误文案。Form 的 validate() 则遍历所有注册进来的 FormItem，汇总所有字段校验结果。

Input 是怎么支持 v-model 的？
    Input 接收 modelValue，内部维护 响应式数据inputValue，原生 input 变化时触发 emit('update:modelValue', value)。
    同时它还监听外部 modelValue 的变化，保证父组件主动修改值时，Input 内部状态也能同步。

Input 和 Form 是怎么联动校验的？
    Input 内部注入 FormItem 的上下文信息，在 input、change、blur 事件里调用 FormItem.validate(trigger)。
    这样 Input 不需要知道 Form 的完整结构，只需要知道“我所在的 FormItem 提供了 validate 方法”。这是一个低耦合设计。

Tooltip 为什么引入 Popper.js？
    Tooltip 的难点不是显示隐藏，而是浮层定位。
    要处理 bottom、top、start、end、偏移、视口边界等问题，自己写容易出错。
    使用 @popperjs/core 创建定位实例，把复杂定位交给成熟库，组件本身主要负责触发方式、显示状态、延迟、防抖和关闭逻辑。

Tooltip 的 manual 模式有什么用？
    manual 模式表示 Tooltip 不自己绑定 hover/click 行为，而是把 show/hide 方法暴露出去，让外部组件控制。
    项目里的 Select 就利用了这一点：Select 自己控制下拉框打开、关闭、筛选和清空逻辑，Tooltip 只负责弹层定位和展示。

Select 组件是怎么设计的？
    Select 是一个组合型组件，它复用了 Input 作为输入框，复用了 Tooltip 作为下拉浮层。
    内部用 states 管理输入值、选中项、hover 状态和 loading 状态。
    点击选项时触发 change 和 update:modelValue，清空时触发 clear。
    筛选方面支持本地 filter、自定义 filterMethod，以及 remoteMethod 异步远程搜索。

Dropdown 和 Select 有什么共同设计？
    它们都复用了 Tooltip 的浮层能力。
    Dropdown 主要是“触发器 + 菜单项 + select 事件”，
    Select 则在此基础上叠加了输入框、选中值、筛选、清空、远程搜索等状态管理。
    这体现了组件库里很重要的思想：把通用浮层能力抽成 Tooltip，再被更复杂的组件复用。

你项目里的 TypeScript 主要体现在哪里？
    第一是 props 类型，比如 Button 的 ButtonType 用联合类型限制取值；
    第二是 emits 类型，比如 Input 明确声明 update:modelValue、input、change 等事件参数；
    第三是 provide/inject 使用 InjectionKey<FormContext> 保证注入上下文有类型；
    第四是 Message 用 Omit<MessageProps, 'id' | 'zIndex' | 'onClose'> 区分内部 props 和用户可传 options。

单元测试主要测了什么？
    项目使用 Vitest + Vue Test Utils，在 jsdom 环境下测试组件渲染和交互。
    比如 Button 测 class、disabled、icon、loading；
    Input 测 v-model 和密码显隐；
    Form 测校验失败和更新后成功；
    交互组件测试里 mock 了 Popper，测试 Collapse、Dialog、Dropdown、Select、Switch、Tooltip 的核心行为。
    这些测试重点覆盖“用户能看到什么”和“用户操作后 emit 什么”。

VitePress 文档站点做了什么？
    docs 目录里按组件组织文档，VitePress 配置了 sidebar，并接入 @vitepress-demo-preview，可以在文档中通过 <preview path="..."> 展示组件 demo。这个对组件库很重要，因为组件库不仅要能用，还要能让别人知道怎么用。

打包发布怎么做的？
    项目使用 Vite 的 lib mode，以 src/index.ts 作为入口，输出 ES 和 UMD 两种格式，并把 Vue 配成 external，避免把 Vue 打进组件库产物里。
    package.json 里配置了 main、module、types 和 exports，同时把 Vue 放到 peerDependencies，这符合组件库发布的基本规范。

这个项目你认为最有技术含量的是哪部分？
    我会选 Message 和 Form。Message 体现了函数式组件调用、VNode 渲染、实例池管理、动画销毁和堆叠偏移计算；Form 体现了 provide/inject 上下文通信、字段注册机制、async-validator 集成和跨组件校验链路。这两块比单纯写 UI 样式更能体现组件库架构能力。

项目还有哪些可以优化的点？
    按需引入可以进一步做成多入口导出；
    浮层类组件可以考虑 Teleport 到 body，避免父级 overflow 或 transform 影响；
    Form 可以支持嵌套路径，比如 user.name；Input 的 validateEvent 默认值需要更明确；
    Tooltip/Select 的独立测试文件目前还比较空，可以补充边界用例；样式系统也可以继续抽成主题 token。

面试官问“你从这个项目学到了什么”，怎么答？
    我理解了组件库不是堆组件，而是要设计稳定 API、类型约束、组件通信、状态管理、样式隔离、文档和测试。
    比如 Button 让我理解 props 到 class 的映射；
    Message 让我理解函数式调用和动态挂载；
    Form 让我理解上下文通信和校验机制；
    打包部分让我理解组件库如何面向使用者发布。

如何让一个组件同时支持PC端和移动端
    响应式布局 + 输入方式兼容 + 交互策略适配
    从样式、事件和交互策略三个层面考虑。
    样式上避免写死宽高，使用响应式布局和媒体查询 —— 使用百分比，设置max-width，根据不同的屏幕宽度写样式
    `
        @media (max-width: 768px) {
            .container {
                padding: 16px;
            }
        }
    `
    事件上不能只依赖 hover，要兼容 click 或 touch；弹层类组件还要处理小屏幕下的位置、溢出和滚动问题，比如移动端可以改成底部弹层或使用 click 触发。
